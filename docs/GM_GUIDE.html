I
Act One

Welcome to the bridge

What this module is, what it deliberately is not, and where it sits between Foundry and the GMhub web app.

In one breath

GMhub-VTT is a Foundry VTT module that two-way-syncs journals only — not scenes, not actors, not combat — between a Foundry world and one GMhub campaign. Your codex (NPCs, Locations, Factions, Items, Quests, Lore) appears in Foundry as JournalEntries. Your active session’s plan does too. You run Live inside Foundry; you press Pull to load and Push to send back. The recap phase still happens in the web app.

Sync is explicit and manual. There is no live websocket, no background polling, no surprise overwrites. The GM decides the direction; the module follows. One world ↔ one campaign, set once in module settings.

§
This manual rides on top of the GMhub Web Manual. Concepts that live on both sides — the visibility model, the codex shape, the four session phases, what an entity “is” — live there. This document references those instead of re-explaining them, so the two read together rather than duplicating each other. Keep that one open if you’re new: GMhub-app/docs/GM_GUIDE.html.

What this module does — and stays out of

Inside the wedge

  • Pull the codex (six entity kinds) + long-form notes + a windowed session slice
  • Push visibility flips, edits, new entries, quick-notes, plan edits, lifecycle events
  • One JournalEntry per entity_kind: NPCs, Locations, Factions, Items, Quests, Lore (six total)
  • One Notes JournalEntry mirroring GMhub’s long-form notes
  • One Session journal per pulled session, under a GMhub Sessions folder
  • Quick-notes via /qn chat command during live sessions

Out of scope — on purpose

  • Maps → Foundry Scenes (Foundry’s scene system is richer)
  • Player characters → Foundry Actors (D&D 5e Actor is canonical)
  • Live websocket / background sync (manual is the contract)
  • Player Foundry users talking to GMhub (only the GM client authenticates)
  • Combat tracker, dice, tokens, compendiums (all native Foundry)
  • Encounter builder, AI assistant (web-app concerns)
i
Conflict policy: direction wins. Pull overwrites Foundry; Push overwrites GMhub. The module never merges. The GM picks the direction. The exception is the dirty-skip on Pull (see Act XI), which protects un-pushed local edits from silent destruction.

Installing the module

One manifest URL pasted into Foundry’s module installer. The download is a few hundred kilobytes; the install finishes faster than your tea steeps.

  1. Open Foundry’s Setup screen

    From the Foundry launcher, pick Add-on Modules → Install Module.

  2. Paste the manifest URL

    Bottom of the install dialog, in the Manifest URL field:

    manifest URL
    https://github.com/b34rblack-glitch/GMhub-VTT/releases/latest/download/module.json

    Hit Install. Foundry pulls the latest release tarball and unpacks it under Data/modules/gmhub-vtt.

  3. Enable the module in your world

    Launch your world → Game Settings → Manage Modules → tick GMhub VTT Bridge → Save. Foundry restarts the world.

  4. Look for the GMhub Sync button

    It appears in the Journal sidebar, next to the Create Folder and Create Entry buttons. That’s your sync surface for everything the module does.

Act Three

Configuration

Four settings, one of which is sensitive. Set once per world; persists across sessions.

Open Game Settings → Configure Settings → Module Settings → GMhub VTT. Four fields:

SettingWhat it does
GMhub Base URL Root URL of the GMhub deployment. Production: https://gmhub.app. Self-hosted or staging: whatever your URL is. No trailing slash needed.
GMhub API Key Your personal access token. Sent as Authorization: Bearer gmhub_pat_…. GM-only — never share with player Foundry users. Stored encrypted in world flags.
Campaign Which GMhub campaign this Foundry world is bound to. After you fill the URL + token, this drop-down lists every campaign you GM. Pick one. One world ↔ one campaign.
Auto-push Off by default. When on, every page edit (text, name, eye toggle) pushes to GMhub immediately. Off honours the manual-sync contract and is the recommended setting.

Minting your API token (the GMhub side)

  1. Go to gmhub.app/account/api-tokens

    The settings page in the web app. Sign in as the GM account that owns the campaign.

  2. Create a new token

    Give it a name (“Foundry — my desktop”) and pick the scopes the module needs:

    campaigns:read entities:read entities:write notes:read notes:write sessions:read sessions:write sessions:secrets

    The last one is GM-only-forever and unlocks the GM Secrets sub-page on session plans. Grant it if you want Secrets to round-trip; omit if you’d rather Secrets stay web-only.

  3. Copy the plaintext — once

    The token shows in plaintext exactly once at issuance. Copy it now or you’ll have to revoke and mint another.

  4. Paste it into Foundry’s GMhub API Key field

    Save settings. The module verifies the token on next Pull / Test connection.

What lands in Foundry · the windowed pull

The module pulls a deliberately narrow window of session data, not the full history:

All prep sessions

Every session that has no started_at. These are the ones you’re still drafting.

The single most-recent recap

The session with the largest ended_at. The last one you closed the book on.

The currently running session

If one exists. The active row. Quick-notes you log in Foundry attach here.

i
Older recaps stay on the web app. If you need to reference session-2 from session-7, open it on gmhub.app in a browser tab. Manual fetch of older recaps is tracked as a future v0.5+ feature; for now the window is the contract.

Visibility ride-along on Pull

Pull respects GMhub’s four-value visibility on every entity and note. The module translates each value into a Foundry per-page ownership level:

GMhub: campaign
▼ maps to
Foundry page ownership: OBSERVER for all players
GMhub: players_only
▼ maps to
Foundry page ownership: OBSERVER for all players
GMhub: gm_only
▼ maps to
Foundry page ownership: NONE for players (GM only)
GMhub: private
▼ maps to
Foundry page ownership: NONE for players (GM only)

Players can never see gm_only or private content through Foundry, regardless of any local ownership tweaks. The mapping is one-way on Pull; on Push the eye-icon state in Foundry maps back (see Act VII).

✦
The visibility model is the heart of GMhub. If this is your first time meeting it, the deeper explanation lives in the Web Manual: GM_GUIDE.html § VIII · Visibility. Read that once and the module’s mapping will feel natural.

The six kind-journals

Each is a Foundry JournalEntry titled by entity kind. Each page inside is one entity. Page name = entity name; page body = entity summary, rendered from GMhub’s ProseMirror-JSON to Foundry-compatible HTML on Pull.

External-ID flags

Every journal and page carries a Foundry flag flags.gmhub-vtt.externalId with the entity’s GMhub UUID. That flag is how the module knows to update in place on subsequent Pulls instead of creating duplicates.

New pages = new entities

Add a new page to the NPCs journal in Foundry — e.g., an NPC you invented mid-session. On Push, the module sees a page with no external-ID flag and creates a new GMhub entity with kind = npc (inferred from the parent journal). GMhub sends back the new UUID; the module writes it into the page’s flag for future updates.

Renaming & re-kinding

Rename a page in Foundry — the entity name updates on Push. Re-kinding (moving a page from NPCs to Items) is not supported; create a new page in the right journal and delete the misplaced one.

Deletes

Delete a page in Foundry — the corresponding GMhub entity is deleted on next Push (idempotent). Delete an entity in GMhub — the corresponding Foundry page is deleted on next Pull (with the dirty-skip exception — see Act XI).

The session journals

Each pulled session lives under the GMhub Sessions folder as one JournalEntry, named YYYY-MM-DD — <title> (date from created_at, so Foundry’s alphabetic sort is chronological). Inside, four pages mirror the four web-app sub-pages:

Foundry pageWhat it carries · who can see it
GM NotesThe main canvas. Default ownership: NONE for players (GM-only) unless you flip the eye to OBSERVER.
AgendaOrdered scene list. Same ownership default as GM Notes; flip if you want players to peek at the schedule.
GM Secrets forever-lockedAlways GM-only. The eye toggle is hidden on this page. Only present if your token has the sessions:secrets scope.
PinnedOne card per pinned entity, with a clickable link into Foundry’s full entity page. Per-pin reasons render as a quoted blockquote line under each card.
⌬
The active session is a pointer, not a journal. Multiple session journals can exist simultaneously (the prep set + last recap + running session). The active session is whichever one the lifecycle buttons (Start / Pause / Resume / End) target. Right-click any session journal → Set as active session to flip the pointer; the existing Pick Session dialog stays as a fallback.

Act Six

Running Live at the Foundry table

When the dice are about to hit, you don’t open gmhub.app — you stay in Foundry. Here’s the surface.

For the conceptual model of what a live session is, see the Web Manual’s Act XI · Going live. The Foundry surface mirrors that model with a different set of clicks.

The pre-flight checklist

  1. Pull the latest

    GMhub Sync → ▼ Pull from GMhub. Latest entities, notes, the prep’d session you’re about to run.

  2. Pick the active session

    Two ways: GMhub Sync → Pick session, or right-click the session journal in the sidebar → Set as active session.

  3. Start the session

    GMhub Sync → ▶ Start. The module hits POST /sessions/{id}/lifecycle with action: start on GMhub’s side; the partial unique index is the source of truth for “single active session per campaign”.

  4. Open the session journal

    The active session journal is your live canvas. Pinned page on the side; GM Notes for the running narrative; quick-notes captured via chat command (next act).

The Pinned page · your reference deck

The Pinned page on the active session is one card per pinned entity. Each card has the entity name, a one-line blurb pulled from the GMhub entity summary, an optional pin reason (rendered as a quoted blockquote line), and a link to the full entity’s page in the kind-journal.

Act Seven

Reveals & visibility from inside Foundry

The eye icon on every Foundry page is your reveal-toggle. Flip it mid-session; on next Push, GMhub gets the same flip.

Foundry shows a small per-page eye icon in every Journal page’s sidebar header. The module hijacks that icon as the visibility toggle. Two states matter:

Eye-OFF (closed) gm_only

Page ownership for players is NONE. They don’t see this page in their journal sidebar at all. Maps back to GMhub gm_only on Push.

Eye-ON (open) campaign

Page ownership for players is OBSERVER. They see the page; they can’t edit it. Maps back to GMhub campaign on Push (and stamps revealed_at if it wasn’t set).

Mid-session reveal · the choreography

  1. The party earns the reveal

    They find the secret room, the misdirected NPC drops their cover, the artefact shows its true edge.

  2. You click the eye on the corresponding journal page

    Open the entity page in Foundry → toggle the eye icon to OPEN. Foundry updates the local page’s ownership instantly — players sitting at the table see the page appear in their sidebar within a second.

  3. The change queues for Push

    The module marks the page dirty. On next Push, the visibility goes to GMhub: PATCH /entities/{id}/visibility with { visibility: "campaign" }, plus PATCH /entities/{id}/reveal if first reveal.

  4. Web-app players see it next time they refresh

    The new entry appears in their codex with a just revealed chip dated to the moment of the Push. Existing recap mention-chips that referenced this entity light up across every session.

The four-value mapping table

Foundry only models two ownership levels per-player (NONE, OBSERVER) for the module’s purposes. GMhub has four. Here’s the round-trip:

GMhub valueOn Pull → FoundryOn Push <- Foundry
campaignOBSERVER for all playersEye OPEN on all players → campaign
players_onlyOBSERVER for all playersEye OPEN cannot disambiguate from campaign — we keep the existing GMhub value if it was players_only; otherwise default to campaign
gm_onlyNONE for all playersEye CLOSED → gm_only
privateNONE for all playersEye CLOSED cannot disambiguate from gm_only — we keep private if it was already private
⌬
To set private or players_only, use the web app. Foundry’s ownership model can’t express those two values without ambiguity. The module preserves them on round-trip but doesn’t let you author them from inside Foundry. They’re rare in practice.

Act Eight

Quick notes during play

The fastest surface for “capture this thought before it’s gone”. One chat command, one line, one Enter.

While a session is live, the module exposes a /qn chat command. Type /qn followed by anything; the line becomes a quick-note attached to the active GMhub session.

Foundry chat · GM
// During session 5, in the Foundry chat box:
/qn Strahd flees up the stairs. Sunsword glows pale silver.
/qn Edda the bartender mentioned the Vistani by name — remember her smell of woodsmoke
/qn Party assumed the silver token was a key. It isn’t. Hold that.

Where the notes go

  1. Queued in Foundry first

    The module stashes each /qn line in a Foundry world flag. A brief network blip never loses a note; the queue is local.

  2. Pushed to GMhub on next Push

    Each queued line goes as a POST /sessions/{id}/quick-notes row, attached to the active session by the active-session pointer.

  3. Lands in the web app’s right-hand timeline

    Same surface a quick-note dropped via the floating + button on gmhub.app would land in. Indistinguishable in the UI from either origin.

The sidebar quick-capture button

If chat-command muscle-memory isn’t your style, the module adds a small + Quick note button to the journal sidebar while a session is live. One field; same backing queue; same destination.

✦
One line per moment. Quick-notes are not paragraphs; they’re bookmarks. The right pattern is a one-line capture every time something noteworthy happens, polished into prose during the post-session recap. Treat /qn like F5 in your save game.
IX
Act Nine

Push · sending it back

When the session ends — or any time you want Foundry’s state to land on GMhub — you click Push. One batched payload; many writes.

What gets pushed

Edits to existing content

Pages with an externalId flag whose content has changed since the last sync. Title renames; body edits; eye-toggle flips.

New entries

Pages without an externalId flag, in any of the six kind-journals or the Notes journal. Module infers the entity kind from the parent journal. GMhub assigns a UUID; module writes it back into the page’s flag for next time.

Visibility / reveal flips

Eye-toggle changes since last sync, in both directions. First-reveal also stamps revealed_at on the GMhub side.

Quick notes

The local /qn queue, drained one row per quick-note, attached to the active session.

Session lifecycle events

Start / Pause / Resume / End actions you triggered from Foundry, targeting the active session. Mirrors the web-app lifecycle.

Plan edits to any session

Not just the active one. Editing a prep session’s Agenda or GM Notes in Foundry pushes back to that session on the next Push.

Deletes round-trip too

Delete a page in Foundry → on Push, the module sends DELETE /entities/{id} (or /notes/{id}). The endpoint is idempotent — a second delete returns 404 cleanly. GMhub’s mention-chip cleanup runs server-side; chips referencing the deleted entity render as plain text.

▼
Push overwrites GMhub. If a co-GM edited @Strahd on the web app since your last Pull, your Push blows their edit away. The conflict policy is direction wins: Pull or Push, never merge. If you’re unsure, Pull first — the module will warn before destroying your local dirty edits (see Act XI).

Act Ten

Session lifecycle from Foundry

Start, pause, resume, end. The four actions you already know from the web app, with the same database constraints.

The Sync dialog has lifecycle buttons that target the active session — whichever session journal you set as active. Each button hits POST /sessions/{id}/lifecycle with the matching action.

ActionEffect on GMhubEffect in Foundry
▶ Start Stamps started_at. Locks the active-session row (the partial unique index). Active pointer set. Quick-note queue starts capturing. Pinned page is treated as live reference.
⏸ Pause Halts the live timer; started_at kept; session stays the active row. Quick-notes still capture. UI shows the timer paused.
▶ Resume Timer ticks again. Same row stays active. UI shows the timer running.
■ End Stamps ended_at. Releases the active-session lock. Recap phase opens on the web app. The session journal gets a recap badge. Active pointer is cleared.

After End · the windowed window slides

Ending a session matters for the next Pull. The session you just ended becomes the new most-recent recap; the previous most-recent recap falls outside the window and (if not dirty) is removed from Foundry on the next Pull. See Act XI for the dirty-skip exception.

▼
Single active session per campaign — enforced at the database. If you press Start on Session 6 while Session 5 is still running, the GMhub server returns 409 single_active_session. The module surfaces this as a clear error toast in Foundry. End or Pause Session 5 first.
i
Ending does NOT auto-Push. Pressing End closes the session row on GMhub but doesn’t flush your queue of edits, quick-notes, and reveals. Click Push explicitly. This is intentional — you may want to review what you’re sending before it lands on the canonical archive.
XI
Act Eleven

Conflicts & dirty edits

Manual sync trades automation for predictability. The cost is a class of conflicts the module handles by warning, not merging. Here’s how to read the warnings.

The two flavours of conflict

A · Web changed under you

You Pulled yesterday. A co-GM edited an entity on gmhub.app this morning. You start writing on the same entity in Foundry, then Push. Your Push wins. The web edit is gone. Direction wins; the module doesn’t merge.

B · You changed locally, now you Pull

You edited an entity in Foundry but haven’t Pushed. You hit Pull. The module would normally overwrite your local work — but the dirty-skip exception protects you. The page is skipped with a warning toast; you Push or manually resolve before the next Pull.

The dirty-skip rule (v0.4.0+)

§
Pull never silently destroys un-pushed local edits. Pages or session journals that carry dirty local changes are skipped on Pull, with a warning toast that lists each one. The GM’s next move is theirs: Push the local edits to keep them, or delete the dirty marker manually if they want the Pull to win.

The dirty-skip applies to both content (pages in kind-journals, Notes journal) and session journals (where a falling-out-of-window session would otherwise be deleted on Pull).

Worked example · the sliding window

  1. You ended Session 4 last week

    Pull at the end of last week brought Session 4 into the “most-recent recap” slot.

  2. This week you ran and ended Session 5

    Session 5 is now the new most-recent recap. Session 4 is outside the window.

  3. You hit Pull

    Module reconciles. Session 5 is updated. Session 4’s journal would be deleted — unless you scribbled a late note on Session 4’s GM Notes page that hasn’t been Pushed yet.

  4. Toast: “1 session journal skipped (dirty)”

    Session 4’s journal sticks around with the dirty-skip warning. You Push — the late note goes to GMhub, the dirty flag clears. Next Pull deletes the journal cleanly.

!
Tombstoning vs delete on Pull was resolved as delete in v0.4.0-α. The module deletes orphaned content on Pull (with the dirty-skip exception), not archives. If you want a paper trail, use the GMhub web app — older recaps remain there forever, even when they fall out of the windowed Pull.

Act Twelve

Troubleshooting

The errors you’re likeliest to see, and what each one is trying to tell you.

"401 invalid_token" on Test connection

Your API key in module settings doesn’t match what GMhub has. Either the token expired (default 90 days, max 1 year), it was revoked, or you copied a partial string. Mint a fresh one at /account/api-tokens and paste again.

"403 missing_scope: entities:write"

The token is valid but doesn’t carry the scope the operation needs. Most commonly missing entities:write or sessions:write. Mint a replacement with the full set (campaigns:read, entities:read/write, notes:read/write, sessions:read/write, and optionally sessions:secrets).

"409 single_active_session" on Start

There’s already a live session for this campaign — either started elsewhere (Foundry on another machine, or directly in the web app), or you started it in this Foundry world but the active pointer drifted. Open the web app and End the stale session, or pick a different active session in the Pick Session dialog.

Pull says “3 pages skipped (dirty)”

You have unpushed local edits on those pages. Push them first to keep them, or open each page and use the Foundry context menu to clear the local dirty marker if you really want the Pull to win. Don’t bypass the warning blindly.

A new page I added in Foundry isn’t in GMhub after Push

Two likely culprits: (1) the page is in a journal that isn’t a recognised kind-journal — it must be in one of the six kind-journals or the Notes journal; (2) the Auto-push setting is on but a previous push errored, and the queue stalled. Check the browser console (F12 → Console) for the error and re-Push.

My Foundry world’s journal sidebar has duplicate journals named “NPCs”

You probably ran the first Pull while the world already had a hand-made journal of the same name. The module created its own NPCs journal (with the GMhub flag) alongside it. Move pages from the hand-made one into the module’s and delete the duplicate — the module-flagged journal is the one that round-trips.

CORS error in the browser console on Test connection

Should not happen against the production gmhub.app deployment — Epic E shipped permissive CORS on /api/v1/*. If you’re hitting a self-hosted or staging deployment, check that the deployment has applyCorsHeaders in its proxy. Foundry typically calls from http://localhost:30000.

429 Rate limited

Per-token sustained limit is 600 requests/minute. The error envelope carries Retry-After with the seconds to wait. The module backs off automatically; if you keep seeing this, your sync size is unusually large or auto-push is hitting an edit storm. Turn auto-push off and rely on manual Push.

The active session pointer keeps moving on its own

Almost certainly because the previous session was ended via the web app or via another Foundry world bound to the same campaign. The module reads the active row from GMhub on Pull and updates the pointer accordingly. Set the pointer explicitly with right-click → Set as active session on the journal you want active.

i
The audit log is your friend. /account/api-tokens on the web app shows every request the token has made, with timestamp, endpoint, and response code. If something feels off, open the audit log and look for the failing request.

The pointer the lifecycle buttons target. Independent of which session journals exist.

Auto-push
Optional setting that pushes every page edit immediately. Off by default.
